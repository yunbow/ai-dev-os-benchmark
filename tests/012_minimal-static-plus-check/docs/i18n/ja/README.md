# テスト 012: 最小静的 + 動的チェック — 両方の組み合わせはチェックのみに勝るか？

## 背景

テスト 011 で動的チェック+修正が +9.9 点（85.3 → 95.2）を達成。本テストの問い: **最小限の静的ガイドライン（3ファイル、~8K tokens）をチェック+修正に加えると、さらに改善するか？**

## 仮説

**H2:** minimal-then-check > baseline-then-check（最小静的+チェックがチェックのみに勝る）

真の場合: 最適アーキテクチャは最小CLAUDE.md + 動的チェック。
偽の場合: CLAUDE.md は不要。チェックのみで十分。

## 条件

| 条件 | ステップ1（生成） | ステップ2（チェック+修正） | 総トークン |
|------|-----------------|--------------------------|:---:|
| baseline-then-check | ガイドラインなし | 同じチェックリスト | ~3K |
| minimal-then-check | 3ファイル（security, validation, project-structure） | 同じチェックリスト | ~11K |

### なぜこの3ファイル？

Test 008/010 のデータで効果が確認された項目を含むファイル:
- security.md: S8, S11 が C のみ Pass
- validation.md: V9, V3 が C のみ Pass
- project-structure.md: D2, D6 が C のみ Pass

## 結果

**H2 確認。96.9 — 過去最高スコア。最小静的 + チェックが最適アーキテクチャ。**

| 条件 | スコア | 差 |
|------|:---:|:---:|
| baseline-then-check（0ファイル + チェック） | 87.4 | — |
| **minimal-then-check（3ファイル + チェック）** | **96.9** | **+9.5** |

3つの静的ファイルがチェックのみでは見逃す7項目を Pass に（S8, V8, V9, N4, D2, T4, P4）。これらはチェックリストだけでは伝えられない**プロジェクト固有パターン**。

### 最適アーキテクチャ（データで確認）

```
CLAUDE.md: 3ファイル（~8Kトークン）  ← プロジェクト固有パターン
  ↓
AI がコード生成
  ↓
/ai-dev-os-check: チェックリスト（~3K）← 包括的検証 + 修正
  ↓
スコア: 96.9/100
```
