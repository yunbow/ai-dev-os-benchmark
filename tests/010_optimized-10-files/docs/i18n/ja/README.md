# テスト 010: 最適化10ファイル — 全修正を統合、ファイル数は増やさない

## 背景

テスト 009 では10→13ファイル（+50%トークン）に拡大したところ、11項目の具体化修正にもかかわらず悪化した。「Less is More」原則は厳選セットにも適用される。

本テストは厳選セットを**10ファイル（~25Kトークン）**に維持 — テスト 001 と同じファイル数 — し、全11項目の修正を既存ファイルに統合する。

## テスト 009 との違い

| | テスト 009 | テスト 010 |
|---|---|---|
| ファイル数 | 13（routing, performance, ui 追加） | **10**（元のセット） |
| トークン | ~36K | **~25K** |
| E4/E5 ルール | routing.md（別ファイル） | **error-handling.md に統合** |
| P4 ルール | performance.md（別ファイル） | **project-structure.md に統合** |
| AC2 ルール | ui.md（別ファイル） | **project-structure.md に統合** |

## 結果

**10ファイルに戻したらパフォーマンス回復。C は 91.4 — 過去最高。**

| 条件 | スコア | 009 との比較 |
|------|:---:|:---:|
| A: ガイドラインなし | **93.1** | +10.4 |
| C: 厳選10（最適化） | **91.4** | **+13.6** |
| Delta (C−A) | **−1.7** | 009: −4.9 → −1.7（改善） |

11項目の具体化修正のうち9項目が全アプリ生成で機能（テスト 009 の13ファイルでは2/11のみ）。トークンを36K→25Kに削減したことで修正が効くようになった。

| 改善（9項目） | まだ失敗（6項目） |
|---|---|
| S11, E4, E5, V3, V8, V9, AC2, N1, N4 | N7, D3, T4, T5, P4, AC3 |
