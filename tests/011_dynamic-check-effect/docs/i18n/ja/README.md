# テスト 011: 動的チェックの効果 — 事後検証は品質を改善するか？

## 背景

これまでの全テストは、コード生成**中に**ロードされたガイドラインの効果を測定した。結果: ベースラインと明確な差なし（C ≈ A）。

本テストは根本的に異なるアプローチを測定: **まず生成し、次にチェックして修正する。** `/ai-dev-os-check` ワークフローのシミュレーション。

## 仮説

> ai-dev-os の価値は生成時のコンテキスト注入ではなく、事後の品質検証にある。

**H1:** A→check > A（動的チェックはベースラインを改善する）
**H2:** A→check > Test 010 の C（チェック+修正は静的ガイドラインに勝る）

## テスト設計

**単一セッション内での2段階プロセス:**

```
ステップ1: ガイドラインなしでアプリ生成（条件Aと同一）
  → step1-generate.md を貼り付け

ステップ2: 同じセッション内で動的チェック+修正を実行
  → step2-check-and-fix.md を貼り付け
  → AI が自分のコードをチェックリスト照合
  → AI が全違反を修正
```

**対照群:** baseline-only（ステップ1のみ、ステップ2なし）

## 条件

| 条件 | プロセス | 比較対象 |
|------|---------|---------|
| baseline-only | ステップ1のみ | = Test 010 条件A |
| baseline-then-check | ステップ1 → ステップ2 | 新条件 |

## 結果

**STRONG GO。動的チェック+修正で +9.9 点 — 全テスト中最大の改善。**

| 条件 | スコア | 改善 |
|------|:---:|:---:|
| baseline-only | 85.3 | — |
| baseline-then-check | **95.2** | **+9.9** |

95.2 は全11回のベンチマークテストを通じて**過去最高スコア**。比較: 静的ガイドラインの最高は C=91.4（Test 010、このベースラインに対して +6.1）。

チェック+修正で改善された7項目（悪化ゼロ）:

| カテゴリ | 項目 |
|---------|------|
| セキュリティ | S9（Math.random → crypto.randomBytes） |
| エラーハンドリング | E4（5つのルートセグメントに error.tsx） |
| バリデーション | V9（日付範囲 .refine()） |
| 命名規則 | N1（kebab-case）、N4（snake_case @@map）、N7（handleTaskDelete） |
| 型安全性 | T4（不要な `as` キャスト除去） |

**主要な洞察:** 静的ガイドライン（CLAUDE.md）は10回のテストでベースラインを明確に上回ったことがない。動的チェック+修正は1回のテストで +9.9。ai-dev-os の価値は**事後の品質検証**にある。
