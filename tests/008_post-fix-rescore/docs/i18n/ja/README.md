# テスト 008: 修正後の再採点 — 具体性の改善は全体スコアを上げたか？

## 背景

テスト 001 では、厳選ガイドラインはベースラインに対して +0.8 しか差がなかった（84.9 vs 84.1）。4項目（V9, N7, T5, P4）は全実行で 0% Pass だった。

テスト 004-007 でこれら4項目をガイドラインの具体化により修正し、各項目で 0% → 100% を達成。本テストは、これらの修正が**全体スコア**にどう影響するかを測定する。

## 目的

1. **スコアマップ**: 改善済みガイドラインで66項目全てのPass/Failを取得
2. **全体への影響**: 厳選条件がベースラインを明確に上回るか？
3. **次のターゲット特定**: まだ0% Passの項目はどれか？

## 条件

| 条件 | コンテキスト | 001 との比較 |
|------|------------|-------------|
| A: ガイドラインなし | 要件のみ（001-A と同一） | ベースラインコントロール |
| C: 厳選10（修正後） | 要件 + 改善済み厳選ガイドライン | V9, N7, T5, P4 修正含む |

## 結果

| 条件 | スコア | 001 との比較 |
|------|:---:|:---:|
| A: ガイドラインなし | **86.1** | 001-A 平均: 84.1（+2.0） |
| C: 厳選（修正後） | **83.2** | 001-C 平均: 84.9（−1.7） |

**4項目の具体化修正だけでは全体スコアは改善しなかった。** AI 出力のばらつき（±4-5点）がシグナルを上回っている。ただし 66項目スコアマップから重要なパターンが判明:

### ガイドラインが効いた項目（C > A）

| ID | 項目 | A | C |
|---|------|:---:|:---:|
| S8 | Email HTML escaping | Fail | **Pass** |
| N4 | DB カラム snake_case | Fail | **Pass** |
| D2 | Features ディレクトリ | Fail | **Pass** |
| D6 | Zod co-located | Fail | **Pass** |
| T3 | Prisma 型活用 | Fail | **Pass** |

### 両条件で未解決（次のターゲット）

| ID | 項目 | 優先度 |
|---|------|:---:|
| E4 | error.tsx 各セグメント | 高 |
| N7 | イベントハンドラ命名 | 中 |
| T5 | exhaustive union check | 中 |
| P4 | Dynamic import | 低 |

**主要な洞察:** 具体化修正は個別タスク（テスト 004-007: 0%→100%）では効くが、全アプリ生成では AI が数百の判断を同時にこなすため完全には転移しない。
